------------------------------------------------------------------------------
1. WHAT THIS PROJECT IS
------------------------------------------------------------------------------
A digital twin for an ion beamline: a stack of 19 metal electrodes that must
steer a beam of 500 Si+ ions (28 amu, 15 eV, launched in a 15-degree half-angle
cone) from a source to a small Detector volume, by choosing the right voltage
on each electrode. 11 electrodes are FIXED (source, detector, grounded parts);
8 are FREE to optimize: electrodes 3, 6, 9, 10, 11, 12, 15, 18 (see optimizer.py
OPTIMIZE dict), each rangeable in [-1000, 1000] V.

Two ways to evaluate a voltage array:
  - REAL SIMION (ground truth, ~5.77s per candidate, dominated by process
    startup + file I/O, not computation -- see section 4).
  - Custom RK4 physics engine in Python (cheap proxy, built this session,
    used to screen many candidates before spending SIMION budget on them).

The intended optimization loop (what the user described wanting, and what
orchestrator.py now implements):
    Optuna sampler suggests candidates
        -> RK4 batch-screens & ranks them (cheap)
        -> top-K run through REAL SIMION (expensive, ground truth)
        -> K real results fed back into the study (study.tell())
        -> repeat, now informed by real SIMION history
    ...until a SIMION-run budget is exhausted or a target score is reached.

4. KEY PERFORMANCE FACTS (all measured, not estimated)
------------------------------------------------------------------------------
  Real SIMION, per candidate:        ~5.77s (5.04s fastadj + 0.73s fly).
                                      Dominated by process startup + loading/
                                      saving the 289MB electrode_.PA0 file,
                                      not the physics computation itself.
  RK4 batched integration:           cost(M configs, N particles, T steps)
                                      ~= T*1.80e-4 + T*2.80e-6*N*M seconds
                                      (fixed per-step cost paid ONCE per
                                      batch; marginal cost per particle).
  RK4 scoring (combined_score):      profiled at ~86% of scoring time inside
                                      the wall-collision check
                                      (WallIndex.distance calls). Reduced
                                      4.2x (8.6s -> ~2.1s for a 50-config
                                      chunk) via wall_check_midpoints=False,
                                      wall_check_stride=3 -- validated to
                                      give IDENTICAL wall_hit_fraction/
                                      combined_score vs the thorough default
                                      across 3 different random batches.
  map_coordinates vs RegularGridInterpolator: 1.6-2.6x faster for this
                                      project's uniform grid.
  Recommended RK4 screening fidelity: 50 particles, 2500 steps (see
                                      cost_planner.py's MIN_VIABLE_STEPS=1000
                                      physics floor and diminishing-returns
                                      finding beyond ~2500).

5. THE BIG FINDING -- read this before running more optimization
------------------------------------------------------------------------------
Across MANY runs this session (RK4-only batches of 42 to 535 candidates,
random voltage draws every time, different samplers, different fidelities,
AND 26+ REAL SIMION runs with random voltages) the result has been
essentially constant:
  - 0 real SIMION hits on the detector, every single time, no exceptions.
  - RK4 combined_score nearly or exactly identical across a whole batch,
    regardless of how different the voltage arrays are (verified: voltage
    arrays differing by up to ~1800V on individual electrodes produced
    IDENTICAL final particle-position statistics, to one decimal place).
  - Real SIMION splat positions cluster at z=1-174mm; the detector sits at
    z=403-407mm -- nowhere close, confirmed independently of RK4.

Two separate causes were found and disentangled:
  (a) BUG, partially fixed: beam_progress_score.py's combined_score() used
      target_scale=30mm (matched the lens/bender FEATURE size, not the
      actual observed miss DISTANCE). Real candidates land 200-350mm from
      the detector; exp(-distance/30) rounds to ~0 for all of them, so the
      one term meant to say "warmer/colder" toward the goal was contributing
      ZERO signal, leaving wall_hit_fraction/lost_fraction (which also
      cluster tightly) to dominate an almost-flat sum. FIXED: target_scale
      raised to 150.0 in beam_progress_score.py, sim_batch.py, and
      orchestrator.py (NOT in voltage_batch_filter.py -- that file is
      superseded, left as-is). Verified after the fix: target_reward is no
      longer silently zero (now ~0.119 typical), and combined_score's
      absolute value shifted meaningfully.
  (b) STILL UNRESOLVED, structural: even after fixing (a), target_reward's
      variance ACROSS candidates in a batch is still tiny (std=0.000002 on
      one measured batch) -- because target_distance itself barely varies
      with random voltage draws. This is NOT a reward-scaling problem, it's
      that the search is not reaching a region where the outcome actually
      changes. Likely root cause: the 4 quadrupole-bender electrodes
      (9, 10, 11, 12) probably need a CORRELATED pattern (opposing pairs)
      to produce real net deflection, and independent per-electrode uniform
      random sampling in an 8-D cube essentially never lands in that thin
      "useful" region by chance.

6. GPSampler: PRACTICAL NOTES
------------------------------------------------------------------------------
  - New dependency: torch (CPU build installed this session via
    `pip install torch --index-url https://download.pytorch.org/whl/cpu`).
  - Hits an OpenMP runtime conflict on this machine -- workaround is setting
    the environment variable KMP_DUPLICATE_LIB_OK=TRUE before running
    anything that imports GPSampler (compare_samplers.py, orchestrator.py
    when using sampler=GPSampler(...)). This is an "unsupported" workaround
    per Intel's own warning -- fine for this exploratory work, worth
    understanding properly before any production use.
  - GPSampler.sample_independent() delegates entirely to a plain
    RandomSampler fallback -- verified by reading its source directly. It
    does NOT consult the GP model for cheap per-parameter draws. Only
    ask() -> suggest_float() -> sample_relative() invokes the real joint
    GP, and that costs ~100-300ms+ per call, growing with trial count
    (roughly cubic, GP refitting). This is why orchestrator.py's GP-seeded
    path pays that cost only n_gp_seeds times per iteration (default 5) and
    generates the rest of the RK4-screening batch via cheap local
    perturbation around those seeds instead.
  - TPESampler.sample_independent() DOES use trial history for its
    per-parameter KDEs and stays flat-cost (~15-34ms) regardless of trial
    count -- this is why it's the default for the cheap M-candidate
    screening step.
  - Observed behavior: GPSampler's ask()-driven suggestions tend to push to
    the parameter bounds (+-1000V) more than TPE's, in this project's data.
    In one real orchestrator run, none of the literal GP seeds ended up
    among the top-K promoted candidates in either iteration -- cheap local
    perturbations around the seeds consistently scored (marginally) better,
    plausibly because boundary-extreme voltages are more wall-hit-prone
    than a milder nearby point.

8. WHERE THIS SESSION LEFT OFF -- next steps under discussion
------------------------------------------------------------------------------
Given finding 5(b) above, three follow-up options were laid out (numbering
matches the conversation, #1 already done):

  #1 [DONE] Widen target_scale (30 -> 150). Necessary but NOT sufficient --
      see section 5(a)/(b). Verified with real before/after measurements.

  #2 [NOT STARTED] Reparameterize the 4 bender electrodes (9,10,11,12) from
      4 independent voltages into ~2 logical "deflection strength" params
      (e.g. bender_z_strength, bender_y_strength) mapped to the 4 real
      voltages through a fixed, physically-derived pattern (which pairs are
      "conjugate" and should move oppositely). Bigger change: touches
      sample_voltage_batch()/sample_voltage_batch_gp_seeded() in both
      sim_batch.py and orchestrator.py, and optimizer.py's objective() if
      the real-SIMION path should use it too. Fixes the STRUCTURAL reason
      random sampling rarely reaches a useful region, not just the reward
      scaling.

  #3 [NOT STARTED] Derive a physically-informed STARTING_POINT (query each
      bender electrode's basis field near the bender region via
      BasisFieldMap.field() to find which sign/combination of V9-V12
      actually pushes the beam toward +z, since the beam needs ~328mm of
      net z deflection from start (z=77) to detector (z=405)) and wire it
      into orchestrator.py's build_study() via study.enqueue_trial(...).
      Smaller, cheaper change than #2; also useful to re-center the cheap
      RK4-screening batch generation around instead of uniform
      [-1000,1000].

  #4 [MENTIONED, NOT STARTED] Space-filling initial sampling (Latin
      Hypercube / Sobol) instead of independent uniform random draws, for
      better 8-D coverage before enough real trials exist to inform TPE/GP
      properly.

KEY INSIGHT for whoever picks this up: #2 and #3 both depend on the SAME
underlying analysis -- sampling each bender electrode's field to determine
its actual deflection direction/magnitude. That analysis has NOT been done
yet. Do it once, then it feeds both options (#3 as a single derived point,
#2 as the pairing pattern for a full reparameterization). The user was
about to decide between #2 and #3 (or do the shared analysis first) when
this handoff was requested instead.

9. THINGS A NEW SESSION SHOULD NOT DO WITHOUT ASKING
------------------------------------------------------------------------------
  - Don't delete anything in legacy/ or the SIMION .PA* files -- they're
    kept as backups/source data, not dead weight, even though large.
  - Don't touch RK4_sim_basis.py or optimizer.py's core logic -- both are
    intentionally kept as unmodified reference/backup versions; new work
    goes in new files (the pattern used consistently this whole session).
  - Don't assume GPSampler's cheap sampling reflects the GP model -- always
    check which code path (sample_independent vs sample_relative/ask) is
    actually being hit before trusting a "the GP suggested X" claim.
  - Don't trust a "combined_score differs" result without checking full
    floating-point precision -- this session found real bugs (flat scores,
    wrong cost-model calibration) hiding behind rounded 3-decimal print
    statements more than once.

ADDENDUM (2026-07-03, follow-up session) -- READ THIS FIRST
================================================================================
The shared bender-field analysis (section 8) was done, and in doing it the
REAL root cause of finding 5(b) was found -- and it was neither the reward
scaling nor the sampling region:

UNIT BUG (fixed): BatchRK4Integrator._derivative computed
  acceleration = q * E / m
with E in V/mm and positions/velocities in mm and mm/s. That formula is
only valid in SI -- in mm-units it under-scales every electric force by
exactly 1e6 (V/mm -> V/m is x1000, m/s^2 -> mm/s^2 is another x1000). The
fields never meaningfully touched the beam, which is why voltage arrays
differing by +-1000V produced trajectories identical to one decimal all
session. FIXED in RK4_sim_basis_batch.py (E_VMM_TO_ACCEL_MM = 1e6).
NOT fixed in RK4_sim_basis.py's RK4Integrator (same bug, but that file is
a kept-as-backup, don't-touch reference -- port the fix if it's ever used
for real again). Its potential() also crashes on a single 1-D point
(reshape bug), found in passing.

Consequences of the fix (all verified by running):
  - The beam leaves the +500V source at ~470 eV (not 15 eV), so DT was
    changed 5e-8 -> 1e-8 in orchestrator.py, sim_batch.py, and the new
    bender_field_analysis.py (5e-8 was ~2.9mm/step at the true speed).
  - With ZERO bender voltage the corrected RK4 flies the beam straight
    into the x~0 wall at z~80 -- matching the real SIMION splats at
    z=1-174mm that section 5 reported. RK4 and SIMION now agree
    qualitatively for the first time.
  - cost_planner.py's MIN_VIABLE_STEPS=1000 "physics floor" and its
    fidelity recommendations were calibrated on the broken dynamics --
    per-step TIMING is still right, but the physics-floor reasoning is
    stale and should be recalibrated before being trusted again.
  - The GPSampler-vs-TPE behavioral comparisons and every RK4 score in
    the existing studies/DBs predate the fix -- treat all persisted RK4
    scores as meaningless. Real-SIMION objective values are unaffected.

DENSE OBJECTIVE -- lever (b) DONE (mean splat distance, best-10% filter):
  orchestrator.py's SIMION objective is no longer the sparse hit count.
  run_simion_candidate() now returns (value, hits, mean_all):
    - value (the objective): mean Euclidean distance from splat position
      to the DETECTOR_REGION box (0 if inside), computed over the
      CLOSEST SPLAT_TOP_FRACTION=10% of ions only -- "how close does the
      beam's leading edge get" -- PLUS HIT_TERM_SCALE_MM=20 * (1 - hit
      fraction). Rationale for top-10%: the all-ion mean is dominated by
      the beam bulk splatting early in the einzel (~324mm out); zero-hit
      configs differed by only ~3.5mm on the first smoke run. Rationale
      for the hit term (saturation fix, caught on the second smoke run):
      the hitting config's 50 closest ions all splat ON the detector
      surface -> best-10% distance is exactly 0.0 with only 5 counted
      hits, i.e. the distance term runs out of gradient precisely when
      transmission becomes the thing to optimize; the hit term hands
      ranking over smoothly (far configs: distance dominates, +~20mm
      constant; saturated configs: 5 hits -> +19.8, 100 -> +16, full
      transmission -> 0). DIRECTION is "minimize" (module-local
      OBJECTIVE/DIRECTION/BAD_SCORE=1e6 override optimizer.py's).
      NOTE: beamline_study_fresh10_splat.db (in legacy/studies) and the
      second 10-run smoke DB predate the hit term -- their stored values
      lack the +20*(1-hitfrac) offset; don't mix-and-match numerically.
    - hits: still computed, stored per-trial in
      user_attrs["simion_hits"]; the NARROW_AFTER_HITS=100 gate and all
      reporting read hits from there, NOT from trial.value.
    - mean_all: all-ion mean distance, stored in
      user_attrs["mean_splat_all_mm"] for diagnostics.
  IMPORTANT: distance-objective studies are minimize; the old
  maximize-hits studies (legacy/studies/*, beamline_study_fresh50/100/
  500.db) must never be appended to with the new code -- fresh_run.py
  now suffixes its DBs "_splat" (beamline_study_freshN_splat.db) so a
  collision can't happen silently.
